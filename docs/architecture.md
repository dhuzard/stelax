# STELAX Architecture

STELAX bridges structured human intent with autonomous task execution and immutable memory.

## Core Flow
1. **Intake**: GitHub Issue Forms (Strict YAML constraint definitions).
2. **Triage**: Normalization queue via native Github actions.
3. **Execution**: Push to Veritas.
4. **Agent Operations**: NanoClaw pulls execution context and spins ephemeral task runners.
5. **Memory**: Completed workflows push artifacts to `<org>-stelax-memory`.

## Downstream Search & Memory
STELAX is designed to operate seamlessly with downstream search tools such as the **Knowledge Agent**. 

**Implementation Notes:**
- The knowledge agent should be pointed directly at `<org>-stelax-memory` via a fresh clone or webhook index hook.
- Do NOT inject the knowledge agent code into STELAX directly. STELAX treats the memory repo as out-of-bounds append-only storage.
- The knowledge agent requires read-only skills for searching file-producing sources (the structured markdown generated by the `exporter`).
